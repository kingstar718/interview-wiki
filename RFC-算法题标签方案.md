## 1. 问题

### 1.1 题解的小标题不携带任何检索信息

159 篇题解里 156 篇的 H2 逐字相同,由校验 L 强制:

```
题目 → 思路 → 代码 → 复杂度 → 边界条件 → 变式 → 易错点 → 面试追问 → 关联题
```

`grep '^## ' problems/` 的产出是 156 份一模一样的列表,信息量为零。

对比八股侧:`gen_index.py` 扫 `interview/` 的真实 H3 能生成 752 行可点击索引,因为那些 H3 是「为什么 HashMap 用红黑树」这类语义标题。**两侧的检索能力不对称,这是本 RFC 要解决的问题。**

### 1.2 现有分类粒度太粗

`topics:` frontmatter 的实际分布:

30 道题共用一个标签,等于没有标签。`42-接雨水` 挂了三个粗桶(双指针 / 数组 / 栈队列),但它真正用到的技术是**对撞指针**、**单调栈**、**前缀和思想**——这三个词一个都没被机器读到。

### 1.3 细粒度词表其实已经写好了,只是没被结构化

13 个套路页的 H2 本身就是一份子套路词表(约 35 条),例如:

- `双指针与滑动窗口.md` → 对撞指针 / 同向双指针 / 滑动窗口
- `栈与队列.md` → 普通栈 / 单调栈、单调队列
- `数组与字符串.md` → 原地操作 / 前缀和与差分
- `图论.md` → DFS-BFS+visited / 拓扑排序 / 并查集

题解和这份词表之间,目前没有任何机器可读的连接。

## 2. 否决的方案:把标签写进小标题

原始设想是在题解里加 `## 标签:单调栈` 这类小节。不采纳,三个理由:

1. **小标题会生成锚点**。`## 标签:单调栈` 产出锚点 `#标签单调栈`,没有任何页面会去链它,污染 `check_index.py` 校验 M 的锚点空间。
2. **会破坏九节固定结构**。九节结构本身有价值:它保证每篇都真的写了复杂度、边界条件、易错点。把元数据塞进正文小节会把这个约束搅浑,校验 L 也要跟着放宽。
3. **标签是元数据,不是内容**。frontmatter 就是元数据的位置,而且 `topics:` 已经在那里了。

补充:原设想的三元组「题目 / 标签 / 关联」里,另外两项**已经存在**——H1 由校验 L 强制为 `42. 接雨水(Trapping Rain Water)`,`## 关联题` 由校验 O 强制带类型前缀(同套路/进阶/基础/易混/知识点)。**唯一缺的是标签。**

## 3. 方案:frontmatter 增加 `techniques:`

### 3.1 数据模型

`topics:` 保留,语义不变(粗分类,决定题目出现在哪个套路页)。新增 `techniques:`,细粒度技术标签,值域受白名单约束。

```yaml
---
topics:
  - 双指针与滑动窗口
  - 数组与字符串
  - 栈与队列
techniques:
  - 对撞指针
  - 单调栈
  - 前缀和
---
```

`techniques` 是**全局扁平词表**,不按套路做命名空间。理由:同一个技术会横跨套路(前缀和既是数组技巧也是 DP 预处理),做成 `套路#技术` 会逼出重复条目。

按决策 A,撞名的词拆开:链表侧 `快慢指针`(判环/找中点),数组侧 `原地读写指针`(读写分工原地覆盖)。

### 3.2.1 被裁掉的 9 条(记录裁剪理由)

| 词 | 实际覆盖 | 处置 |

| `差分` | 2(1094 / 1109) | 并入 `前缀和与差分` |
| `单调队列` | 1(239) | 并入 `单调栈与单调队列` |
| `二分答案` | 1(69) | 并入 `二分边界` |
| `最短路` | 1(787) | 删,787 归 `图遍历visited` |
| `区间DP` | 1(312) | 删,312 归 `二维DP` |
| `状态压缩` | 0 | 删,题库里没有 |
| `记忆化搜索` | ≤2 | 删,与 `线性DP`/`二维DP` 重合 |
| `位掩码` | 2(78 / 338) | 删,78 归 `回溯框架`,338 归 `线性DP` |
| `嵌套结构栈` | — | 更名拆分为 `栈队列设计` + `表达式与嵌套栈` |

**阈值定为 2(决策 G)。** 取 3 会逼出两个坏合并:`快速选择` 被并进 `堆TopK`,丢掉「这题能用快速选择」(quickselect 平均 O(n) vs 堆 O(n log k) 是两种解法);`BST中序` 被并进 `递归返回值设计`,而「中序有序」根本不是「递归返回什么」,语义上就是错的。这两个词各只有 2 题但语义干净,予以保留。

阈值真正该防的是 0~1 题的词(`状态压缩` 0 题、`单调队列`/`二分答案`/`最短路`/`区间DP` 各 1 题),它们已被删除或合并。

### 3.2.2 三个套路的标签没有增量信息

| 套路 | 题数 | 技术词 | 问题 |
|---|---|---|---|
| 字典树 | 6 | `Trie前缀树` × 6 | `techniques` ≡ `topics` |
| 回溯 | 13 | `回溯框架` × 13(+ `剪枝` × 6) | 主词覆盖全集 |
| 二分查找 | 8 | `二分边界` × 8 | `techniques` ≡ `topics` |

这三个套路里,细粒度标签不产生新的检索维度。**这不是缺陷,是这些套路本身就只有一个模式。** 保留标签是为了词表完整(否则这些题 `techniques` 为空,违反校验 S),但不要指望它们提供检索价值。

### 3.2.3 `模拟构造` 是必要的兜底词

回填时发现:66 加一、118 杨辉三角、54 螺旋矩阵、48 旋转图像这类题**没有可归纳的技术**,它们的技术就是「按题意模拟」。校验 S 要求 `techniques` 非空,所以必须有兜底词。

风险:兜底词会变成垃圾桶。**约束:`模拟构造` 覆盖数超过 8 题就要复查**,说明有题被偷懒归类了。这一条写进 CONTRIBUTING.md。

### 3.2.4 P2 试点的三条修正(2026-07-10 实测 21 篇后回写)

试点 `双指针与滑动窗口`(11)+ `栈与队列`(12,去重 21 篇)暴露三个问题,均已修正:

1. **`辅助栈` 这个词不成立,改名 `栈队列设计`。** 155 最小栈确实是辅助栈,但 225(队列实现栈)、232(双栈实现队列)是「用一种结构模拟另一种」,不是辅助栈;而 155 单独成词只有 1 题,低于阈值 2。三题的共性是**设计题**,与哈希表页的 `哈希+双向链表` 对称。词表仍是 35 条(等量替换)。

2. **同一个技术词可以被多个套路页声明。** 白名单是「这一页展示哪些分组」,不是「这个词归谁所有」。283 移动零、75 颜色分类的 `topics` 含双指针,但 `原地读写指针` 原本只归数组页,严格模式下它们在双指针页会永远落进未标注。修正:双指针页也声明 `原地读写指针`,数组页也声明 `对撞指针`(11/15/42 在数组页同样该进这一组)。全局词表 = 13 页声明的**并集**,去重后仍是 35 条。

3. **半迁移的页面比平铺还难读**,渲染规则加了一道闸。11/15/42/283 的 `topics` 里也有数组与字符串,给它们打标签会让数组页立刻进入分组模式:27 题里 4 题有组、23 题堆在「未标注技术」下。修正:**一个套路页只有在它的题全部标注完之后才切换成分组显示**,否则整页保持平铺。这样回填可以按套路逐个推进,不影响其余套路页,`未标注技术` 这个分组永远不会出现在页面上(只作为闸门的判据)。

试点结果:双指针页 11/11、栈与队列页 12/12 全部标注,两页已切换为分组显示;其余 11 页保持平铺,一个字没动。

### 3.3 套路页如何声明自己有哪些技术词

`techniques` 的白名单需要一个权威源。三个做法:

| 做法 | 说明 | 评价 |
|---|---|---|
| (a) 改套路页 H2 为短名 | H2 从「对撞指针:利用单调性排除整段区间」改成「对撞指针」 | ❌ 损失表达力,现在的长 H2 是内容资产 |
| (b) 约定 H2 格式为「短名:说明」,取冒号前缀 | 脚本按冒号解析 | ❌ 现有 H2 格式不统一(「二分的本质是找分界点,不是找有序数组里的值」没有冒号) |
| (c) **套路页 frontmatter 显式声明** | 见下 | ✅ **推荐** |

```yaml
# algorithms/双指针与滑动窗口.md
---
title: 双指针与滑动窗口
kind: [pattern]
techniques:
  - 对撞指针
  - 滑动窗口
---
```

(c) 的好处:白名单不绑定 H2 文案,套路页正文怎么改都不会碰坏校验;13 个套路页的 `techniques` 并集就是全局白名单,天然去重。

### 3.4 校验与生成

**新增校验 S(词表合法)**:每篇题解的 `techniques:` 非空,且每个值都在「13 个套路页 frontmatter 声明的 techniques 并集」里。与现有校验 F(拦 `topics:` 空值/拼写错误)完全同构,实现可复用。

按决策 C,校验 S 取**严格版**:`techniques` 里的词必须属于该题 `topics:` 声明的某个套路。不满足就把套路加进 `topics:`(它本来就可多值)。

**新增校验 T(词表成色)**:每个技术词至少覆盖 N 道题。低于阈值说明词立不住,应合并或删除。这一项防止词表膨胀成一堆没人用的词。**N = 2(决策 G)**:0~1 题的词说明词立不住;2 题但语义干净的词予以保留。

**改造 `gen_topics.py`**:`## 已解题目` 从平铺列表改为按技术词分组。

```markdown
<!-- gen:problems:begin 由 scripts/gen_topics.py 生成，勿手编 -->
### 对撞指针
- [11. 盛最多水的容器](11-container-with-most-water.md)
- [15. 三数之和](15-3sum.md)
- [42. 接雨水](42-trapping-rain-water.md)

### 滑动窗口
- [3. 无重复字符的最长子串](3-longest-substring-without-repeating-characters.md)
...
<!-- gen:problems:end -->
```

按决策 D,分组结果留在**单一生成块**内(如上,块内用 H3 分组),不拆开插到讲解 H2 底下——后者阅读体验更好,但生成块散落多处、`splice` 逻辑复杂、校验 R 要重写,而讲解正文里本来就已经点名了代表题。

校验 R(套路视图一致)随之升级为「分组结果与 `techniques:` 一致」。

### 3.5 检索能力的兑现

这是整件事的目的,写清楚免得做完发现没用上:

- `grep -l 'techniques:.*前缀和'` → 直接列出所有前缀和的题(现在做不到)
- `outline.py --grep` 可以扩展成按技术词检索
- 套路页的 `## 已解题目` 分组后,「同一套路举一反三」从口号变成可点的清单
- 校验 T 会主动暴露「某个技术只有 1 道题」→ 要么词不成立,要么题库缺题,两种都是有用的信号

## 4. 回填策略

**159 篇要回填 `techniques:`,这是本方案的主要成本。**

**纯 grep 自动回填走不通。** 实测候选词在题解正文的命中文件数:

| 词 | 命中 | | 词 | 命中 |
|---|---|---|---|---|
| 递归 | 45 | | 单调栈 | 6 |
| 贪心 | 33 | | 拓扑排序 | 4 |
| 回溯 | 24 | | 记忆化 | 2 |
| 滑动窗口 | 14 | | **对撞指针** | **0** |
| 前缀和 | 11 | | **虚拟头节点** | **0** |
| 快慢指针 | 7 | | **位掩码** | **0** |
| 并查集 | 7 | | **状态压缩** | **0** |

四个技术词在 159 篇正文里**一次都没出现**——它们只活在套路页。同时「递归」命中 45 篇,但那多半是「递归」两个字被顺口提到,不代表该题的技术是递归。**信噪比两头都不行。**

建议的三步:

## 8. 关联决策(上一轮讨论,本 RFC 不执行,但方向一致)

结论:**算法是算法,八股是八股,`概念/` 不该覆盖算法。**

支撑事实:9 个概念页里只有 `摊还` 和 `树` 有算法侧入链,其余 7 个入链全在 `interview/` 内部。而且 `摊还` 的 5 条算法侧链接全是「整页引用」形态(正文里一句「这也是摊还分析,参见」),没有一条落到具体小节——是延伸阅读,不是承重结构。

更根本的理由:**算法侧本来就有自己的概念层——13 个套路页**。它和 `概念/` 是同构的(原子 → 抽共性 → 反向视图)。让 `概念/` 去覆盖算法,是在已有概念层上再盖一层。

本 RFC 落地后,算法的概念层会因为 `techniques:` 而变得更实(套路 → 技术 → 题,三级),`概念/` 更没有理由伸进算法侧。

待办(另开条目,不在本 RFC 范围):

- `CLAUDE.md` 删除「算法套路 ↔ 八股机制的汇合点」表述,`概念/` 定位改为「跨 interview 分类的知识节点」
- 校验 P 的域只按 interview 分类计,`algorithms` 不计域(核实过:`摊还`、`树` 去掉算法域后仍达标,9 个概念一个都不用回收)
- 算法 → 概念 的单向链接保留但不计入入场券:**把「可以链」和「必须链」分开**
- 拆 `概念/树.md`:「二叉搜索树」那节讲的是解题入口(中序有序),应移进 `algorithms/二叉树.md`;`树.md` 只留介质定价 + 红黑/AVL/B/B+/跳表选型

## 8.5 P3 的三条推翻(2026-07-10,6 个 subagent 并行回填后回写)

P0 定的词表按**题目标题**估算覆盖数,没读正文。P3 让 6 个 agent 各自读 159 篇的 `## 思路` 和 `## 代码` 后,推翻了三条:

### 推翻一:校验 T(每词 ≥N 题)不该存在

被独立证伪三次:N=3 逼出 `快速选择`→`堆TopK`、`BST中序`→`递归返回值设计` 两个语义上就错的合并;降到 N=2 保下 `快速选择` 后,实测它自己只剩 1 题(agent 判定 offer40 的正文里 quickselect 只有一句带过、无代码);四个 agent 独立提出的新词(`中心扩展`/`对顶堆`/`多路归并`/`节点映射`)全是 1 题。

**词表条数不是成本(回填才是),拿条数下限去卡语义精确是拿错了东西当约束。** 校验 T 废除,改为 `gen_topics.py` 打印审计:仅覆盖 1 题的词高亮提示,不阻断。

### 推翻二:决策 C 的严格模式假设了「每个 topic 都贡献技术」,这是错的

`01-两数之和` 的 topics 是「哈希表 + 数组与字符串」,但它的技术**纯粹是哈希**——「数组」只是它作用的数据类型,不是技术来源。`56-合并区间` 同理(技术是 `区间排序`)。这类**挂靠型 topic** 在严格模式下无解。

不用改校验。P2 已确立的「同一个词可被多个套路页声明」正是解法:数组页也声明 `哈希查表` 和 `区间排序`,01 与 56 就在数组页落进正确的分组。同理二叉树页声明 `回溯框架`(收 113 路径总和 II)、字典树页声明 `哈希查表`(收 720)。

### 推翻三:词表 35 → 44,位运算是重灾区

位运算 6 题里只有 2 题是 `异或自反`。P0 把 `位掩码`(2 题)按阈值删掉,还把 78 塞给 `回溯框架`、338 塞给 `线性DP` —— 全错。补 `位掩码枚举`(78)、`逐位构造`(338 位DP / 371 加法器 / 461 popcount)。

9 个新词:`线性扫描维护状态`(53/121/152,Kadane 式滚动变量)、`逐位构造`(3)、`位掩码枚举`、`最短路`(787 是 Bellman-Ford,不是 DFS/BFS+visited)、`多路归并`(23)、`对顶堆`(295)、`节点映射`(138)、`跳表设计`(1206)、`中心扩展`(5)。
两处改名:`图遍历visited` → `图遍历`(797 是 DAG,压根没有 visited)、`哈希+双向链表` → `哈希+辅助结构`(380/381 用的是 ArrayList,一根链表都没有)。

### 附带确立:`techniques` 以「代码实际实现的解法」为准

`720-词典中最长的单词` 思路首列 Trie,代码却是 `Arrays.sort` + `HashSet`;`5-最长回文子串` 正文主解法是中心扩展,DP 只提一嘴。检索标签的契约是「按标签找过来能看到对应的代码」,所以以代码为准。720 改标 `哈希查表`,并登记内容待办:补 Trie 实现。
