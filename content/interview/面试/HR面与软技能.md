## 一、HR 常见问题与回答策略

### 自我介绍

**考察点**：提炼总结能力、沟通逻辑、与岗位的匹配度。

**结构模板（3 分钟）**：

1. **我是谁**（学历/工作年限 + 岗位方向）
2. **核心经验**（几段经历、什么公司、负责什么方向）
3. **突出成绩**（1～2 个量化成果，数字最有说服力）
4. **求职方向**（为什么看机会 + 为什么投这家）

**示例**（后端 3 年）：

> 我叫 XXX，有 3 年后端开发经验，前两年在小红书负责用户增长方向，最近一年在美团到店做订单系统。在美团期间主导了订单分库分表改造，将高峰写入 TPS 从 2000 提升到 8000，QPS 稳定在 1.2 万以上。看机会是因为想接触更高并发场景，贵公司在电商和支付领域的技术深度是我很向往的。

**避坑**：
- 不要背简历 —— 面试官手上有简历，重点讲简历上看不到的取舍和思考
- 不要讲流水账 —— 选 1～2 个亮点深讲，其他一笔带过
- 不要超过 5 分钟 —— 控制不住长度说明提炼能力不够

---

### 离职原因

**考察点**：稳定性、职业诉求、是否会在新公司重演。

**回答原则**：
- **讲客观，不主观**：组织架构调整、业务方向变化、通勤太远
- **讲未来，不抱怨**："想接触 XXX 方向"而不是"那边技术太落后"
- **讲事实，不编造**：背调可查，谎言一戳就破

**常见安全答案**：
| 场景 | 安全回答 |
|------|---------|
| 业务调整 | 部门业务线收缩/整体裁撤 |
| 发展瓶颈 | 现有岗位成长空间有限，想挑战更高难度的技术场景 |
| 技术栈不匹配 | 团队主要维护老系统，希望在新项目中使用更现代的技术栈 |
| 通勤/家庭 | 搬家后通勤超过 1.5 小时，客观上无法持续 |
| 创业失败/裁员 | 坦诚说明，突出经验和反思，不回避 |

**避坑**：
- ❌ "领导太蠢"、"加班太多"、"薪资太低"
- ❌ "公司政治斗争严重"
- ❌ 空窗期太长不说原因（主动说明空窗期做了什么：学习/旅行/思考方向）

---

### 期望薪资

**考察点**：自我定位、谈判空间、对薪资架构的了解。

**策略**：
1. **不要先报价** —— 尽量让 HR 先出范围
2. **了解架构** —— 薪资 = 月薪 × 月数 + 绩效/年终 + 股票/期权 + 补贴
3. **锚定能力** —— 用经验/成果锚定上限，而非数字本身

**话术**（HR 追问时）：

> 我目前总包在 45W 左右。对标这个岗位的要求——我有三年高并发经验、主导过亿级数据分库分表——结合贵司的薪资范围，我期望 50～55W 的总包。当然，如果培养体系好、成长空间大，薪资可以再聊。

**关键信息**：
- 了解目标公司的薪资结构（base 占比、年终惯例）
- 了解行业水平（Offershow、脉脉、一亩三分地）
- 心理底价 → 目标价 → 理想价，分三层准备

**避坑**：
- ❌ 直接说一个死数字，不留余地
- ❌ "薪资都行，先让我进去" —— HR 可能真给最低档
- ❌ 过早透露当前薪资（HR 问可以说"不方便透露，更关注总包是否能匹配岗位价值"）

### 职业规划

**考察点**：稳定性、目标感、是否和公司发展一致。

**三层结构**：

| 阶段 | 时长 | 目标 |
|------|------|------|
| 短期 | 0～1 年 | 快速熟悉业务、融入团队、独立承担模块开发 |
| 中期 | 2～3 年 | 深耕技术领域、成为核心贡献者、能带新人 |
| 长期 | 3～5 年 | 成为技术专家或架构师，能在某一方向输出影响力 |

**示例**：

> 短期来说，我的首要目标是尽快熟悉业务和现有技术体系，争取在半年内独立负责一个核心模块。中期我希望在分布式系统或者数据库方向持续深耕，成为团队在这个方向的专家，能主导技术方案设计。长期来看，我希望不仅自己能做，还能带人和沉淀方法论，往架构师方向发展。

**避坑**：
- ❌ "没想过"、"走一步看一步"
- ❌ 太虚："三年当 CTO"
- ❌ 和岗位/公司方向不匹配（应聘后端说想转产品）

---

### 优缺点

**考察点**：自我认知是否清晰、与岗位是否匹配。

**优点**：贴靠岗位胜任力模型

| 后端岗位看重 | 对应优点表达 |
|-------------|------------|
| 技术深度 | 对源码有好奇心，喜欢追根究底 |
| 工程素养 | 代码整洁、注重测试、有 Code Review 习惯 |
| 团队协作 | 善于沟通，能在技术方案中兼顾各方诉求 |
| 抗压能力 | 能快速响应线上问题，紧急情况下保持冷静 |
| 学习能力 | 关注社区动态，有 Side Project 或技术博客 |

**缺点**：选非核心能力，且已经有改进动作

| 缺点（安全） | 为什么安全 | 改进动作 |
|-------------|-----------|---------|
| 公开演讲紧张 | 后端开会多但演讲少 | 在组内分享中主动练，已做 3 次 |
| 代码写得太快， review 容易被挑 | 节奏问题，非能力问题 | 现在写完自检一遍再提 MR |
| 对业务结果敏感度不足 | 后端常见问题 | 主动和 PM 对齐业务指标 |
| 英语口语不够流利 | 大部分团队够用即可 | 每周参加 English Corner |

**避坑**：
- ❌ "我太完美了"（遮丑式回答被识破）
- ❌ "没有缺点"（自我认知差）
- ❌ 说了致命缺点（比如"粗心"、"逻辑不好"、"不擅长合作"）

### 加班看法

**考察点**：工作投入度、是否排斥加班文化。

**标准姿势**：

> 我现在正是拼搏事业的阶段，如果项目需要，加班完全没问题。不过我的做事风格是先想清楚再动手，通过合理的方案设计和技术手段来减少不必要的工作量。如果团队普遍长期超负荷运转，我也会主动推动流程优化或者技术债清理——帮团队提效，这对公司和我是双赢。

**要点**：
- 第一句表态：能接受加班
- 第二句表态：但也不盲目加班，讲究效率
- 第三句表态：如果是长期低效加班，愿意推动改进

**避坑**：
- ❌ 直接拒绝："不加班，到点就走"（除非你真的不缺 offer）
- ❌ 表忠心过度："没事我天天加"（HR 反而担心你效率低）
- ❌ 反问："你们加班多吗？"（终面谈 offer 时再问）

### 为什么选我们公司

**考察点**：是否做了功课、意向度、求职动机。

**三维回答**：

| 维度 | 准备方向 | 示例 |
|------|---------|------|
| 业务 | 公司主营业务、市场地位 | "贵公司在本地生活领域市场份额持续增长" |
| 技术 | 技术栈、技术博客、开源项目 | "看到贵司技术博客关于 MySQL 优化的文章，很有启发" |
| 文化 | 价值观、团队风格 | "我认同工程师文化的理念，技术方案有争论空间" |

**示例**：

> 我很早就关注贵公司的技术博客，尤其是订单系统那篇分享对我启发很大。加上贵司的业务正处于快速增长期，技术挑战和成长空间都很大，和我的职业规划也很匹配。

**避坑**：
- ❌ "我也不太了解，就投了"
- ❌ 背官网介绍（HR 每天听几十遍，要加个人真实感受）
- ❌ 只夸不接地气（要把公司和自己的经历/目标连接起来）

### 反问面试官

**考察点**：你的思维层次、是否真的对这个机会感兴趣。

**按面试阶段准备问题清单**：

**技术面（组长/技术负责人）**：

| 问题 | 说明 |
|------|------|
| 这个岗位的核心技术挑战是什么？ | 了解技术方向和难度 |
| 团队现在的技术债主要在哪些方面？ | 考察技术负责人是否务实 |
| 技术选型的决策流程是怎样的？ | 了解技术话语权 |
| 新人的 onboarding 和成长路径是怎样的？ | 关心培养体系 |
| 目前团队的业务和技术指标是什么？ | 了解团队目标 |
| 如果我能加入，您希望我半年内有什么产出？ | 明确预期 |

**HR 面/终面**：

| 问题 | 说明 |
|------|------|
| 这个岗位所在的团队目前有多少人？分工是怎样的？ | 了解团队结构 |
| 公司对员工的培养方式有哪些？ | 关心成长 |
| 这个岗位的晋升周期和标准是怎样的？ | 关心发展 |
| 您觉得什么样的人在这个团队里能做得比较出色？ | 反向了解文化 |
| 公司对于这个部门的定位和资源投入是怎样的？ | 了解重视程度 |

**避坑**：
- ❌ "没什么想问的"（= 对岗位没兴趣）
- ❌ 问太早（一上来就问食堂班车年终奖，印象分直线下降）
- ❌ 问太敏感（其他 offer 薪资、公司财务数据）
- ❌ 问能百度到的（"贵公司主营业务是什么"）

## 二、简历编写指南

### STAR 法则

每条项目经历按 STAR 四要素组织：

| 要素 | 含义 | 示例 |
|------|------|------|
| **S**ituation | 项目背景、业务场景 | 到店订单系统日订单量 200 万，MySQL 单库写瓶颈 |
| **T**ask | 你的职责和目标 | 负责订单分库分表方案设计，目标 TPS 提升 3 倍 |
| **A**ction | 具体做了什么 | 采用 user_id hash 分 64 库，引入 ShardingSphere，灰度切换 |
| **R**esult | 可量化的结果 | TPS 从 2000 → 8000，P99 从 120ms → 35ms |

**每段项目写 3～5 条，优先级从左到右递减**。

### 技术栈描述

**分级法**（比全列 + 柱状图更清晰）：

| 级别 | 含义 | 写法 |
|------|------|------|
| ⭐⭐⭐ | 精通，有源码级理解 | Java（JUC/AQS 源码）、MySQL（InnoDB 引擎） |
| ⭐⭐ | 熟练，有丰富实战 | Redis（缓存/分布式锁）、Kafka（消息/流处理） |
| ⭐ | 了解，有基础认知 | Kubernetes、Go |

**原则**：
- 只列自己能**深入聊 15 分钟以上**的技术
- 不列"会用 = 精通"（面试问深了马上露馅）
- 按方向分组（语言 / 框架 / 中间件 / 数据库 / 工具链）

### 项目描述模板

**好**：

> **订单分库分表改造**（2024.03 - 2024.06）
> - 背景：订单日增 200 万，单库写入 TPS 达到 1800，逼近 MySQL 瓶颈
> - 职责：独立完成方案设计、核心代码实现、灰度上线
> - 动作：对比 hash/range/一致性哈希，选 user_id hash 64 库；引入 ShardingSphere 做透明分片；设计双写校验 + 灰度切流兜底
> - 结果：TPS 8000+，P99 降低 70%，零数据不一致事故

**不好**：

> 订单系统优化（2024）
> - 对数据库做了分库分表
> - 使用 ShardingSphere
> - 性能提升很多

**量化技巧**：

| 指标 | 可量化的角度 |
|------|-------------|
| 性能 | TPS/QPS 提升、延迟降低、资源节省 |
| 稳定性 | 可用性 SLA、故障减少次数、MTTR |
| 业务 | 日活/订单量/转化率提升 |
| 效率 | 发版频率、排查时间减少、成本降低 |
| 规模 | 数据量、节点数、服务数 |

## 面试追问地图

| 主问题 | 关键点 | 下一层追问 |
|--------|--------|-----------|
| 自我介绍 | 一句话标签 + 2 个亮点 | 为什么离职 / 为什么选我们 |
| 离职原因 | 客观原因 + 诉求匹配 | 空窗期在做什么 / 上家绩效 |
| 期望薪资 | 薪资架构 + 能力锚定 | 当前薪资 / 其他 offer |
| 职业规划 | 短期 → 中期 → 长期 | 为规划做了什么准备 |
| 优缺点 | 优点贴岗位 / 缺点带改进 | 你同事/领导怎么评价你 |
| 为什么选我们 | 业务 + 技术 + 文化 | 竞品对比 / 还投了哪些公司 |
| 反问面试官 | 技术/团队/发展 | 见上方问题清单 |

> 面试本质是 **匹配度** 考察：能力匹配 × 意愿匹配 × 稳定性匹配。HR 面的每一个问题都围绕这三个维度展开，回答时想清楚对方想听什么再开口。
