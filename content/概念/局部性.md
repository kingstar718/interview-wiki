# 局部性

> **一句话**：程序访问的地址不是随机撒开的——刚碰过的还会再碰（时间局部性），碰过的旁边也会被碰（空间局部性）。整个存储层级、以及大半个「为什么数组比链表快」，都建立在这个经验事实上。

## 为什么这些是同一个模式

局部性是一条**关于真实程序的经验观察**，不是定理。但正因为它几乎总成立，硬件和系统才敢做同一件事：**猜你接下来要什么，提前搬到更快的地方。**

- **时间局部性** → 缓存有意义。刚访问的数据留着，很可能马上再用。
- **空间局部性** → **成批搬运**有意义。取一个字节，顺手把它周围的 64 字节（一个 cache line）、或 4KB（一页）一起搬过来。

一旦你的访问模式违反局部性，这两条优化就全部反噬：每次访问都 miss，每次都白搬一整批。**性能不是慢一点，是塌方式地慢一到两个数量级。**

这解释了一批看起来毫无关系的现象：

| 现象 | 局部性在哪 |
|---|---|
| 数组遍历远快于链表遍历 | 数组元素连续，一个 cache line 装好几个 |
| Kafka 顺序写盘比随机写内存还快 | 顺序访问让预读和页缓存全部命中 |
| TLB 需要存在 | 页表项的访问高度集中 |
| 多路归并 k 很大时改用堆 | 减少对每条链表头部的反复扫描 |

## 数据结构:数组 vs 链表

「数组随机访问 O(1)、链表 O(n)；链表插入 O(1)、数组 O(n)」——这是复杂度的答案，但**不是性能的答案**。

真实情况是：即便在链表中间插入元素这种链表的主场，只要你得先遍历过去找到位置，数组往往仍然更快。因为遍历时数组是连续预取，链表是每个节点一次可能的 cache miss。

`LinkedList` 在实际工程里几乎从不比 `ArrayList` 快，原因就在这里——不是复杂度算错了，是复杂度模型假设「每次内存访问代价相同」，而这个假设早就不成立了。见[集合框架](集合框架.md#arraylist-和-linkedlist-的性能差异)与[数据结构与算法](数据结构与算法.md#数组-vs-链表)。

## 系统:顺序写与页缓存

Kafka 的高吞吐里，「顺序写」排在零拷贝前面。磁盘顺序写能到几百 MB/s，随机写只有几 MB/s——差距来自寻道，而顺序访问让寻道次数趋近于零。

页缓存则同时吃两种局部性：读的时候预读后续页（空间），命中的页留在内存里（时间）。所以 Kafka 干脆不自己做缓存，直接把这件事交给操作系统。

稀疏索引也是同一个逻辑：**只索引到段，剩下的靠顺序扫**——因为顺序扫在有局部性的介质上几乎免费。见[消息队列](消息队列.md#kafka-日志存储segment-和稀疏索引怎么按-offset-找到一条消息)。

## 易错点:局部性是假设，不是保证

**它可以被输入摧毁。** 哈希表的平均 O(1) 假设哈希均匀，局部性优化假设访问集中——两者都会被特定输入打穿。链式哈希表在遍历时局部性极差，这也是为什么开放寻址在只读场景常常更快。

**「批量搬运」在没有局部性时是纯损耗。** 随机访问一个巨大数组，每次取 4 字节却搬进 64 字节，浪费 15/16 的带宽。这就是列式存储要按列存的原因：分析型查询只读少数几列，行式存储每次搬进来一整行全是无用数据。

**大对象会挤掉别人的缓存。** 一次扫描 10GB 的冷数据，会把整个 CPU 缓存和页缓存冲刷干净，拖慢所有其他线程。这类「缓存污染」在批处理任务和在线服务混部时是真实事故。
