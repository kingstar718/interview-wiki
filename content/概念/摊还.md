# 摊还

> **一句话**：单次操作最坏很贵，但贵操作出现的频率被前面大量廉价操作「买单」了，于是一整串操作的平均代价仍是常数。看单次会误判，必须看序列。

## 为什么这些是同一个模式

摊还分析回答的不是「这次操作多贵」，而是「连续做 n 次总共多贵」。判断一个场景能不能用摊还，问三句话：

1. 有没有**偶发的昂贵操作**（扩容、rehash、批量出栈）？
2. 这个昂贵操作的代价，是不是**正比于此前积累的廉价操作次数**？
3. 昂贵操作发生后，计数器是否**清零重来**？

三个都是，总代价就能被摊平。ArrayList 扩容复制 n 个元素，但这 n 个元素是此前 n 次 O(1) 的 `add` 一个个塞进去的，复制的账早就在那 n 次里预付了；扩容之后容量翻倍，下一次扩容要再等 n 次 `add`。所以 `add` 是摊还 O(1)，最坏 O(n)。

**「摊还 O(1)」和「平均 O(1)」不是一回事。** 平均是对输入分布取期望，换个输入可能就退化；摊还是对操作序列的确定性上界，**任何**输入下连续 n 次操作的总代价都被卡住。哈希表的平均 O(1) 会被恶意碰撞打穿，ArrayList 的摊还 O(1) 不会。面试里把这两个混着说，是很容易被追问穿的地方。

## 数据结构里的摊还

**动态数组扩容**是最标准的例子：`ArrayList.add()` 摊还 O(1)、最坏 O(n)，扩容按 1.5 倍增长，见[集合框架](集合框架.md#arraylist-扩容机制)。关键是**按倍数**而非按固定量扩容——按固定量扩容会退化成 O(n)，因为昂贵操作的频率不再随规模下降。

**红黑树的近似平衡**：插删触发的旋转次数摊还 O(1)，这正是它比严格平衡的 AVL 更适合读写均衡场景的原因，见[树](树.md#平衡树红黑树与-avl)。

**Redis 渐进式 rehash**：不一次性迁移整张哈希表，而是在每次增删查改时顺带迁移一个桶，把一次 O(n) 的停顿摊到后续的 n 次操作里，见 [Redis](Redis.md#哈希表渐进式-rehash)。这是摊还思想被**主动用作延迟控制手段**——不是分析出来的性质，是设计出来的性质。

## 算法题里的摊还

一旦看到「内层还有 while，但整体是 O(n)」，答案基本都是摊还。证明的套路是**给每个元素记账**，说明它一生最多被处理常数次：

- **单调栈**：每个下标入栈、出栈各一次，所以尽管内层 while 可能连续弹出多个元素，总弹出次数不超过总入栈次数 n。见 [739. 每日温度](739-daily-temperatures.md)。
- **用栈实现队列**：每个元素一生最多经历「压入 inStack、弹出 inStack、压入 outStack、弹出 outStack」四步，`pop` 最坏 O(n) 但摊还 O(1)。见 [232. 用栈实现队列](232-implement-queue-using-stacks.md)。
- **滑动窗口**：左指针总共只前进 n 次，内层 while 的总执行次数被左指针的行程锁死。见 [76. 最小覆盖子串](76-minimum-window-substring.md)。

## 易混:批处理摊薄固定开销

「pipeline 把网络 RTT 摊薄到每条命令上」听起来也叫摊，但**它不是摊还**。

| | 摊还分析 | 批处理摊薄固定开销 |
|---|---|---|
| 贵的东西 | 偶发的昂贵操作（扩容） | 每批一次的固定成本（一次 RTT） |
| 便宜的东西 | 大量廉价操作 | 批内的多个单元 |
| 机制 | 廉价操作为将来的昂贵操作预付 | 一份固定成本被批内单元分担 |
| 反例 | 按固定量扩容就退化 | 批太大则延迟变长 |

前者是**分析方法**（一个已有算法的真实复杂度是多少），后者是**设计手法**（主动攒批来降低单位成本）。Redis pipeline、Kafka 的 `linger.ms` 攒批、MySQL 的 `rewriteBatchedStatements` 都属于后者，见 [Redis](Redis.md)、[消息队列](消息队列.md)。

两者共享「总代价除以操作数」的直觉，但把它们混为一谈，会在「为什么按固定量扩容不行」这种追问上答不出来。
