# 幂等

> **一句话**：同一个请求执行一次或多次，对业务产生的最终效果相同。它不是「不重复执行」，而是「重复执行也没关系」——因为在不可靠的网络里，你根本无法阻止重复。

## 为什么这些是同一个模式

分布式系统里的重试、消息队列的重投、用户的重复点击、客户端超时后的再发——它们制造重复的方式不同，但你的应对**只能是同一种**。

原因是一条无法绕开的事实：**超时不等于失败。** 请求发出去，超时了，你不知道服务端是没收到、执行了一半、还是执行完了但响应丢在回程。三种情况在客户端看来一模一样。

于是只剩两条路。要么把「是否已执行」这个状态查清楚——但查询本身也会超时，无穷递归。要么**让重复执行无害**，也就是幂等。

所以幂等不是一个可选的优化，是分布式系统里**唯一可实现的正确性策略**。这解释了为什么它同时出现在接口设计、消息消费、分布式事务、微服务重试四个毫不相干的地方。

## 天然幂等与需要构造的幂等

查询、删除（`DELETE /order/1` 删一次和删十次结果一样）、以及所有「设置为某个绝对值」的操作（`SET status = PAID`）天然幂等。

危险的是**相对操作**：`balance -= 100`、`count++`、「插入一条流水」。它们执行两次结果就不同。HTTP 的语义约定也建立在这条区分上——`PUT` 是完整替换所以应当幂等，`PATCH` 是部分更新，是否幂等取决于你怎么写，见 [API设计与接口治理](API设计与接口治理.md#创建接口如何保证幂等)。

**把相对操作改写成绝对操作，是最廉价的幂等手段**，而且不需要任何额外存储。能改写就别上幂等键。

## 实现层次

从便宜到贵，能用上一层就别用下一层：

1. **改写成绝对操作**：状态机条件更新 `UPDATE ... SET status='PAID' WHERE status='UNPAID'`，重复执行时影响行数为 0，天然安全。
2. **数据库唯一约束**：用业务唯一键（订单号）挡住重复插入。这是**最终防线**，因为它由数据库保证原子性，不依赖应用逻辑正确。
3. **幂等键 + 结果表**：客户端提交稳定的 `Idempotency-Key`，服务端存「幂等键 + 请求摘要 + 处理状态 + 最终响应」。重复请求且内容一致时**返回原结果**，内容不一致则拒绝复用同一个键。见 [API设计与接口治理](API设计与接口治理.md#创建接口如何保证幂等)与[分布式系统](分布式系统.md#如何设计幂等-key)。
4. **Token 一次性消费**：适合表单防重提，但只挡得住「同一个页面提交两次」，挡不住网络层重试。

消息队列的「消费幂等」是第 2、3 层的应用——见[消息队列](消息队列.md#消息队列为什么会有重复消费怎么解决)。

## 易错点:幂等键选错了等于没做

幂等键必须**稳定代表同一次业务意图**，通常是「业务类型 + 用户/订单 + 请求序号」。

三个高频错法：

- **用时间戳或 UUID 当键**：客户端重试时生成了新的键，服务端认不出这是同一次意图，幂等形同虚设。**键必须由发起方在第一次请求前就定好，重试时原样带回。**
- **只存键不存请求摘要**：同一个键配上不同的请求体（比如金额被改了），服务端会把第二次当重复而返回第一次的结果，等于静默丢单。所以要比对摘要，不一致就报错而不是复用。
- **只存键不存结果**：重复请求时只知道「处理过了」，却答不出「结果是什么」，只能返回一个含糊的成功——客户端拿不到订单号。**幂等的完整语义是「重复请求返回与首次相同的响应」**，不只是「不重复执行」。

## 易混:幂等不是去重，也不是恰好一次

**去重**是把重复消息丢掉；**幂等**是允许它进来但不产生副作用。前者需要一个可靠的「见过没有」的集合（有状态、有过期问题），后者可以由业务写法本身保证。

**「恰好一次（exactly-once）」在端到端上不可实现**——这是分布式系统的基本结论。工程上说的 exactly-once，实际是「至少一次投递 + 消费侧幂等」的组合效果。所以当有人说「我们用了 Kafka 的 exactly-once」，追问下去总会落到幂等上。
