# 间接层

> **一句话**：把「逻辑单元 → 物理资源」的直接绑定，改成「逻辑单元 → 中间单元 → 物理资源」。分布和归属立刻从固定事实变成可调的自由度——这是分布式路由、虚拟内存、消息分区在做的同一件事。

## 为什么这些是同一个模式

直接绑定的系统里，「谁归谁管」由一个函数写死。`hash(key) % N` 决定 key 归哪个节点，虚拟地址直接就是物理地址，消息直接投给某台 broker。这种系统有两个共同的病：

**一变全变。** N 从 3 变成 4，几乎所有 key 换家——缓存集群扩一台机器，命中率瞬间归零，等于自己制造了一次雪崩。

**粒度太粗。** 归属的最小单位就是物理资源本身，于是均匀性没法调、迁移没法控、权重没法设。一个节点下线，它管的东西整块砸给邻居。

加一层中间单元之后，这两条同时松开：逻辑单元先映射到大量的中间单元（虚拟节点、哈希槽、页、分区），中间单元再映射到少量物理资源。前一层的映射是稳定的，后一层的映射是**可以随时改的一张表**。

于是你得到三个自由度，而它们在每个系统里都以同样的面目出现：

| | 中间单元 | 均匀性 | 迁移粒度 | 权重 |
|---|---|---|---|---|
| 一致性哈希 | 虚拟节点（160/物理节点） | 大数定律铺开 | 一段小弧 | 多派虚拟节点 |
| Redis Cluster | 哈希槽（16384 个） | 槽数远多于节点 | 一个槽 | 多分槽 |
| 虚拟内存 | 页 | 按页离散分配 | 一页 | — |
| Kafka | 分区 | 分区数远多于 broker | 一个分区 | 多分分区 |

**中间单元的数量必须远大于物理资源的数量**，否则这一层白加——这是全部四个系统都遵守的同一条设计约束。

## 分布式:虚拟节点与哈希槽

一致性哈希用**算法**隐式决定归属：虚拟节点散布在哈希环上，key 顺时针找到第一个就是家。哈希槽用**元数据**显式决定归属：一张 16384 项的槽表，谁管哪个槽写在表里。

两者中间单元的作用完全一样，区别只在这张映射表是算出来的还是存出来的。**一致性哈希用算法省掉元数据，哈希槽用元数据换控制力。** 数据库类系统搬数据要原子性和节奏控制，所以 Redis Cluster 选了显式槽表。

详见[分布式系统](分布式系统.md#一致性哈希为什么需要虚拟节点)与 [Redis](Redis.md#cluster-为什么是-16384-个槽槽迁移时请求怎么路由)。

## 操作系统:虚拟内存

进程看到的是连续的虚拟地址空间，实际落在哪些物理帧上由页表决定。中间单元是**页**。

好处和上面逐条对应：进程不必关心物理内存是否连续（均匀性）、换页以页为单位（迁移粒度）、多个虚拟页可以映射到同一物理帧（这就是写时复制和共享内存的基础）。见[操作系统](操作系统.md#虚拟内存与物理内存)。

## 消息系统:分区

topic 是逻辑概念，分区是中间单元，broker 是物理资源。生产者按 key 哈希到分区，分区再分配到 broker。

扩容时搬的是分区而不是 topic，这就是为什么「分区数不是越多越好，但必须远多于 broker 数」。见[消息队列](消息队列.md#kafka-为什么不支持读写分离分区数是不是越多越好)。

## 代价

间接层不是免费的，面试里问到这一层才算真的理解：

**多一次查表。** 虚拟地址每次访问都要走页表，所以必须加 TLB 做缓存；TLB miss 的代价大到需要多级页表 + 硬件走表来摊平。

**中间单元本身要被管理。** 16384 个槽的归属信息得在集群里传播（Redis 用 gossip），这是一致性哈希不需要付的账。

**它不解决热点。** 虚拟节点解决的是**节点分布倾斜**，解决不了**单 key 访问热点**——明星商品那个 key 落在哪个虚拟节点上，它就把哪台机器打挂。这是最常见的追问陷阱：以为加了间接层就万事大吉。

> 计算机科学的所有问题都可以通过增加一个间接层来解决，除了「间接层太多」这个问题本身。
