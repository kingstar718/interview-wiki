# 内容贡献规范 — interview-wiki

新增或修改 `content/` 下的内容前先读本文。目录/命名/链接的硬约定见 [CLAUDE.md](./CLAUDE.md)，本文只管**内容本身怎么写**。

## 工作流

```text
1. 领任务：打开 TODO.md，从「待办」挑一项（或先把新想法登记进去）
2. 验证：python3 scripts/outline.py --grep <考点词> 确认未覆盖/深度不足，避免重复劳动
3. 定位：python3 scripts/outline.py <目标文件> 看章节结构，决定新小节插入位置
4. 写内容：按下方模板写入对应专题文件
5. 同步：追问地图行 → 相关篇目互链；`indexes/知识点索引.md` 由 `scripts/gen_index.py` 从各篇目 H3 自动生成,改完跑脚本刷新,勿手编条目
6. 自检：python3 scripts/gen_index.py 刷新知识点索引 → python3 scripts/check_index.py 必须通过
7. 更新 TODO.md：该项移到「已完成」，附 commit 短哈希
```

## 知识点小节模板

新增小节放入对应专题文件的对应章节。结构如下（带 ⭕ 的小节可选，其余必填）：

```markdown
### 问题标题（用面试官原话的问法，如"RR 到底解决幻读了吗？"）

⭕ 频次 ★★★★ · 难度 🟡 · 高频：字节/美团

**是什么**：一句话定义；多概念对比用表格。

**为什么这么设计**：动机与取舍，回答"不这么做会怎样"。

⭕ **源码**：核心机制附简化源码/伪代码，标注来源版本与文件名，
   只留主干逻辑并用注释解释（Java 用 ```java，Redis/内核用 ```c）。

⭕ **对比同类**：与同类方案的表格对比（跳表 vs B+树、REQUIRES_NEW vs NESTED）。

**常见追问**：2~3 个，用"问题 → 答题方向"格式，覆盖追问地图里承诺的下一层。

⭕ **通用概念**：把该机制抽象为跨系统模式（环形缓冲区、多数派、上下文传递策略），
   并互链体现同一模式的其他篇目。
```

写作要求：

- **答题要点导向**：告诉读者"先说什么再说什么、哪句是区分背题和理解的关键"，不是百科全书式罗列
- 算法题另有固定模板与 H1/元数据行约定，见下方「算法题解规范」

## 结构硬约定（interview/ 专题文件，校验 G/H/I 拦截）

- **小节标题 = 稳定语义 ID**（与"文件名 = 稳定 ID"同一原则）：
  - **禁止数字编号开头**（`### 1. xxx`）——编号是位置型 ID，中间插一节就要重编、产生噪音 diff，还为锚点链接埋雷
  - 用**问法式语义标题**（`### 缓存预热：冷启动时怎么避免打挂数据库？`）
  - **发布后不轻改**：标题就是这节的 ID 和锚点，改标题视同改文件名
- **章节序号保留**（`## 一、Redis 基础`）：表达阅读顺序，正文可引用"见第一章"
- **追问地图固定置顶**：每个专题第一个章节必须是 `## 面试追问地图`（不带章号），随后是 `[← 返回知识点](知识点索引.md)`；「面试问题深挖指南」豁免
- **元数据行**（可选）：标题下一行，格式 `频次 ★★★★ · 难度 🟡 · 高频：字节/美团`
  - 三段均可省略，但**出现即校验**：★ 为 1~5 个、难度只用 🟢/🟡/🔴、公司在 CLAUDE.md 约定清单内
  - 不再使用旧写法 `（难度：Hard）`（已批量迁移）

## 算法题解规范（algorithms/，校验 J/K/L 拦截）

- **H1 格式**：`# 题号. 中文题名（English Title）`，如 `# 1. 两数之和（Two Sum）`——难度不进 H1
- **元数据行**（必填，H1 下一行）：`频次 ★★★★★ · 难度 🟢 · 高频：全厂`
  - 语法同 interview 元数据行（校验 I），公司多值用 `/` 分隔，`全厂` 是合法值
  - **题解元数据行是难度/频次/公司的权威源**，`算法题索引.md` 题单表是它的视图（校验 K 比对，同"目录是权威源、枢纽是视图"的关系）；索引表还承担进度视图职责（题名带链接 = 已有题解）
- **固定小节，顺序不变**（校验 L）：题目 → 思路 → 代码 → 复杂度 → 边界条件 → 变式 → 易错点 → 面试追问 → 关联题
  - 「关联题」存量回填期间暂为可选（TODO 有回填批次），回填完成后转必填
- **变式 vs 关联题的分工**：变式管"本题改条件怎么办"（有序版/返回所有解/空间约束），不需要跳走；关联题管"跳出去练什么"，四类按需有则列：
  - `同套路`：同一技巧的兄弟题（快慢指针：141/142/876）
  - `进阶`：难度递进链（206 → 92 → 25）
  - `易混`：看着像但解法不同的题，一句话点破差异
  - `知识点`：与 interview 篇目互链（LRU ↔ 集合框架、环检测 ↔ JVM 引用计数）
- **关联题链接约定**：已收录的题**必须带链接**（提到题号却不链接会被校验 J 拦截）；未收录的只写题号题名，等收录后回补——欠链会被 J 自动发现，关联网络随收录趋于完整

## 修改与整合规范

- **重复考点**：一处深讲，他处一句话 + 链接。选"更系统/更深"的一方做权威（如会话安全归安全认证、CAP 归分布式系统）
- **跨域视角不同的重复**（死锁之于 Java/OS、零拷贝之于 OS/Kafka）：**保留两份，只加互链**，不删内容
- 互链写在被引用内容的紧邻位置，格式 `见[篇名](篇名.md)` 或 `见[[篇名]]`（纯文件名全库唯一匹配，见 CLAUDE.md 链接约定）
- 收编时保留原处的"答题速记"（3 行以内），不要留一个光秃秃的链接
- 改分类/文件名前先看 CLAUDE.md 的命名与索引约定（`interview/<分类>/` 目录结构是权威源）

## 存量内容整改规范（docsify 遗留内容升级，2026-07 启动）

迁移自 docsify 时期的旧内容与新体系并存，整改按以下规则执行（每篇一次整改批次，登记在 TODO.md）：

1. **章 = 主题，不是合并批次**：消灭"`## 一、XXX` + `## 二、XXX 补充`"式拼接——"补充"章的小节按主题归入正式章节，章号只表达阅读顺序。同一考点在一篇内只出现一次（跨篇重复走「修改与整合规范」）。
2. **拆掉 docsify 三件套外壳**：`核心概念速查` / `高频面试题` / `最佳实践` 这层壳不保留——速查表无增量信息则删，有信息则并入对应小节；壳下的问题小节提级为正常 H3。
3. **浅小节升级**：只有"快答"两三行的小节，升级为知识点模板结构（至少补"为什么"和"常见追问"各一条）；暂时无力深写的保持简短，但登记 TODO，不留"深答省略/详见本文档"式的空头引用。
4. **事实基线（版本口径）**：
   - 全站以 **JDK 8 为对照基线、17/21/25 为现代口径**，提及版本敏感的实现细节必须标注版本（如 String 的 char[]→byte[] 是 JDK 9 分界）
   - 预览/孵化特性必须写明 JEP 状态与落地版本，**禁止把未落地提案写成已发布**
   - 工具/库推荐避开有安全争议或已停维护的选择（如 fastjson 1.x → Jackson/fastjson2）
5. **追问地图 = 正文目录**：地图每行的"必讲关键点/下一层追问"必须能在正文找到承接小节；正文没有的，要么先补正文，要么该行暂缓上地图（缺口登记 TODO）。
6. **元数据行补频次**：旧式只有 `难度 🟡` 的行合法但信息量不足，整改时对 ★★★★ 以上的高频小节补齐 `频次` 段；把握不准的保持 `难度` 单段，不硬编。

## 新增小节后的同步清单

| 同步点 | 位置 | 说明 |
|--------|------|------|
| 追问地图 | 专题文件顶部表格 | 新增/修改对应行，"下一层追问"列要和正文的常见追问呼应 |
| 知识点索引 | `content/indexes/知识点索引.md` | 由 `gen_index.py` 从篇目真实 H3 + github-slugger 锚点自动生成,勿手编;改完 H3 跑 `python3 scripts/gen_index.py` 刷新 |
| 互链 | 相关篇目 | 正文提到其他专题的机制时就近加链接 |
| 校验 | 仓库根目录 | `python3 scripts/check_index.py` 退出码为 0 |
| 待办 | `TODO.md` | 完成项移入「已完成」，附 commit |
