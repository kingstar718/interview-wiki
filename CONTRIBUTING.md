# 内容贡献规范 — interview-wiki

新增或修改 `content/` 下的内容前先读本文。目录/命名/链接的硬约定见 [CLAUDE.md](./CLAUDE.md)，本文只管**内容本身怎么写**。

## 工作流

```text
1. 领任务：打开 TODO.md，从「待办」挑一项（或先把新想法登记进去）
2. 检索：确认未覆盖/深度不足，避免重复劳动（见下方「一、检索」）
3. 定位：python3 scripts/outline.py <目标文件> 看章节结构，决定新小节插入位置
4. 写内容：按对象类型走对应流程（见下方「二、新增」）
5. 收尾：跑四条生成/校验命令（见下方「四、收尾」）
6. 更新 TODO.md：该项移到「已完成」，附 commit 短哈希
```

# 操作手册

下面按**任务**组织（检索/新增/修改）；再往下的章节按**对象**组织（知识点/算法题/套路/概念），是各自的写作模板与硬约定。

## 一、检索：动手前先查

| 要找什么 | 怎么找 |
|---|---|
| 八股考点是否已覆盖 | `python3 scripts/outline.py --grep <考点词>` —— 标题命中 + 正文命中（标注所在小节）|
| 某篇目的章节结构 | `python3 scripts/outline.py <文件名关键词>` —— 标题树 + 行号 |
| 哪些题用了某个算法技术 | `python3 scripts/outline.py --tech 单调栈` —— 列出该技术词下全部题解 |
| 全部技术词及覆盖题数 | `python3 scripts/outline.py --tech`（不带参数）|
| 某个知识点在哪一节 | `content/indexes/知识点索引.md` —— 真实 H3 + 锚点，点击直达小节 |
| 某道题属于哪些套路 | 题解 frontmatter 的 `topics:`；或看套路页的 `## 已解题目` |
| 某个跨域概念被谁引用 | `概念/<概念名>.md` 的 `## 出现在哪里`（按小节分组的反链）|

**永远先 `--grep` 再写。** 重复考点的处置规则见下方「修改与整合规范」——是加互链，不是再写一遍。

## 二、新增：按对象分五条路

### 新增八股小节（interview/ 的一个 H3）

写作模板见「知识点小节模板」。收尾：同步追问地图行 → 相关篇目互链 → 跑 `gen_index.py`。

### 新增八股篇目（interview/<分类>/<新篇>.md）

1. 文件放进对应分类目录（**目录结构是分类的唯一权威源**）
2. 在 `quartz.ts` 的 Explorer 排序表 `ORDER` 里登记位置——登记的是**页面 H1 标题**，不是文件名
3. 首个 H2 必须是无章号的 `## 面试追问地图`（校验 G）
4. 文件名全库唯一、语义命名、无数字前缀（校验 B/C）

### 新增算法题解（algorithms/problems/<题号>-<slug>.md）

1. 文件名用题号做稳定 ID：`42-trapping-rain-water.md`；剑指题用 `offer40-` 前缀
2. H1 为 `42. 接雨水（Trapping Rain Water）`；剑指题无官方英文名，只要中文题名（校验 L）
3. 紧跟 H1 的元数据行必填，且它是难度/频次/公司的**权威源**（两个索引是它的视图，校验 K/N）
4. 九个小节固定顺序，**且都不能是空壳**——校验 L 同时检查小节名、顺序与内容非空
5. frontmatter 两级归属都必填：`topics:`（粗套路）+ `techniques:`（细技术词），校验 F/S
6. `## 关联题` 每条带类型前缀：同套路/进阶/基础/易混/知识点（校验 O）
7. 同步 `indexes/算法题索引.md` 与（若进 Top40）`indexes/高频题目索引.md`

### 新增技术词（44 词的词表）

词表的权威源是**套路页自己的 frontmatter `techniques:`**，13 页的并集即全局词表。

1. 在对应套路页的 frontmatter `techniques:` 里加这个词（**声明顺序 = 页面上的分组展示顺序**）
2. 给相关题解的 `techniques:` 打上该词
3. 跑 `gen_topics.py` 刷新分组，跑 `check_index.py` 过校验 S

**一个词可以被多个套路页声明**（`哈希查表` 同属哈希表、数组与字符串、字典树三页）。白名单的语义是"这一页展示哪些分组"，不是"这个词归谁所有"。

新词不需要覆盖多少道题才够格——**词表条数不是成本，回填才是**。`gen_topics.py` 会把只覆盖 1 题的词高亮出来供人判断，但不阻断。

### 新增跨分类概念（概念/<概念名>.md）

入场券是**入链覆盖 ≥2 个 interview 分类**（校验 P，导航链接不算）。`algorithms` 不计域——算法有自己的概念层（套路页），见 CLAUDE.md 的「分域原则」。

`## 出现在哪里` 由 `gen_concepts.py` 生成，勿手编（校验 Q）。

## 三、修改：改了什么就跑什么

| 改动了 | 必须跑 | 因为 |
|---|---|---|
| interview/ 篇目的 H3 标题 | `gen_index.py` | 知识点索引是 H3 的视图 |
| 题解的 `topics:` / `techniques:` | `gen_topics.py` | 套路页「已解题目」是它的视图 |
| 任何指向 `概念/` 的链接 | `gen_concepts.py` | 概念页「出现在哪里」是反链的视图 |
| 题解的元数据行 | 手工同步两个索引 | 元数据行是权威源，索引是视图（校验 K/N）|
| 分类归属 | 先移动文件，再同步索引底部的文件清单 | 目录是权威源 |

**生成物绝不手编**，手改会被下一次生成覆盖。三份生成物的形态不同：

| 生成物 | 形态 | 脚本 | 校验 |
|---|---|---|---|
| `indexes/知识点索引.md` | **整篇生成**（不含标记块，全文都是产物）| `gen_index.py` | —（`--check` 检漂移）|
| `概念/*.md` 的 `## 出现在哪里` | `<!-- gen:refs:begin -->` 标记块内 | `gen_concepts.py` | Q |
| `algorithms/*.md` 的 `## 已解题目` | `<!-- gen:problems:begin -->` 标记块内 | `gen_topics.py` | R |

小节标题是**稳定语义 ID**，发布后不轻改——改了会打断所有指向 `文件.md#锚点` 的链接（校验 M 拦死链）。

## 四、收尾：四条命令，顺序不能乱

```bash
python3 scripts/gen_index.py      # 刷新知识点索引（读 interview/ 的 H3）
python3 scripts/gen_concepts.py   # 刷新概念页「出现在哪里」（读全库反链）
python3 scripts/gen_topics.py     # 刷新套路页「已解题目」（读题解 topics/techniques）
python3 scripts/check_index.py    # 19 项校验，退出码非 0 即有问题
```

三个 `gen_*.py` 都支持 `--check`：只检测漂移不改文件，CI 用这个。

## 知识点小节模板

新增小节放入对应专题文件的对应章节。结构如下（带 ⭕ 的小节可选，其余必填）：

```markdown
### 问题标题（用面试官原话的问法，如"RR 到底解决幻读了吗？"）

⭕ 频次 ★★★★ · 难度 🟡 · 高频：字节/美团

**是什么**：一句话定义；多概念对比用表格。

**为什么这么设计**：动机与取舍，回答"不这么做会怎样"。

⭕ **源码**：核心机制附简化源码/伪代码，标注来源版本与文件名，
   只留主干逻辑并用注释解释（Java 用 ```java，Redis/内核用 ```c）。

⭕ **对比同类**：与同类方案的表格对比（跳表 vs B+树、REQUIRES_NEW vs NESTED）。

**常见追问**：2~3 个，用"问题 → 答题方向"格式，覆盖追问地图里承诺的下一层。

⭕ **通用概念**：把该机制抽象为跨系统模式（环形缓冲区、多数派、上下文传递策略），
   并互链体现同一模式的其他篇目。
```

写作要求：

- **答题要点导向**：告诉读者"先说什么再说什么、哪句是区分背题和理解的关键"，不是百科全书式罗列
- 算法题另有固定模板与 H1/元数据行约定，见下方「算法题解规范」

## 结构硬约定（interview/ 专题文件，校验 G/H/I 拦截）

- **小节标题 = 稳定语义 ID**（与"文件名 = 稳定 ID"同一原则）：
  - **禁止数字编号开头**（`### 1. xxx`）——编号是位置型 ID，中间插一节就要重编、产生噪音 diff，还为锚点链接埋雷
  - 用**问法式语义标题**（`### 缓存预热：冷启动时怎么避免打挂数据库？`）
  - **发布后不轻改**：标题就是这节的 ID 和锚点，改标题视同改文件名
- **章节序号保留**（`## 一、Redis 基础`）：表达阅读顺序，正文可引用"见第一章"
- **追问地图固定置顶**：每个专题第一个章节必须是 `## 面试追问地图`（不带章号）；「面试问题深挖指南」豁免
- **元数据行**（可选）：标题下一行，格式 `频次 ★★★★ · 难度 🟡 · 高频：字节/美团`
  - 三段均可省略，但**出现即校验**：★ 为 1~5 个、难度只用 🟢/🟡/🔴、公司在 CLAUDE.md 约定清单内
  - 不再使用旧写法 `（难度：Hard）`（已批量迁移）

## 算法题解规范（algorithms/，校验 J/K/L 拦截）

- **H1 格式**：`# 题号. 中文题名（English Title）`，如 `# 1. 两数之和（Two Sum）`——难度不进 H1
- **元数据行**（必填，H1 下一行）：`频次 ★★★★★ · 难度 🟢 · 高频：全厂`
  - 语法同 interview 元数据行（校验 I），公司多值用 `/` 分隔，`全厂` 是合法值
  - **题解元数据行是难度/频次/公司的权威源**，`算法题索引.md` 题单表（校验 K）与 `高频题目索引.md` A 表（校验 N）都是它的视图，同"目录是权威源、枢纽是视图"的关系；两张表还承担进度视图职责（题名带链接 = 已有题解）。改题解元数据后必须同步这两张表，公司列只比集合不比顺序（按主考公司在前书写）
- **固定小节，顺序不变**（校验 L）：题目 → 思路 → 代码 → 复杂度 → 边界条件 → 变式 → 易错点 → 面试追问 → 关联题
  - 「关联题」存量回填期间暂为可选（TODO 有回填批次），回填完成后转必填
- **变式 vs 关联题的分工**：变式管"本题改条件怎么办"（有序版/返回所有解/空间约束），不需要跳走；关联题管"跳出去练什么"。关联题**是知识图谱的边**，每条必须以下列类型前缀开头（校验 O 拦截白名单外的写法）：
  - `同套路`：同一技巧的兄弟题（快慢指针：141/142/876）
  - `进阶`：难度递进链（206 → 92 → 25）
  - `基础`：本题依赖的前置题，`进阶` 的反向（919 依赖 102 的 BFS 模板）
  - `易混`：看着像但解法不同的题，一句话点破差异
  - `知识点`：与 interview 篇目或 `概念/` 节点互链（LRU ↔ 集合框架、单调栈 ↔ 摊还）
  - 反向边不必写：Quartz 的 backlinks 会自动呈现"谁引用了我"，每条边只在一处声明
- **关联题链接约定**：已收录的题**必须带链接**（提到题号却不链接会被校验 J 拦截）；未收录的只写题号题名，等收录后回补——欠链会被 J 自动发现，关联网络随收录趋于完整

## 算法套路节点规范（algorithms/，校验 F/R 拦截）

`algorithms/` 下只有两类文件：`algorithms/problems/<题号>-slug.md` 是原始题解扁平池，`algorithms/<套路名>.md`（13 个，语义命名，无数字前缀）是套路节点——同一套路能解决表面上毫不相关的题，套路节点讲的是"为什么"，题解讲的是"这一题怎么做"。

- **归属 = frontmatter，不是目录**：题解在 frontmatter 声明 `topics: [套路名, ...]`（可多值——一题可能同时属于多个套路，如"接雨水"横跨数组、双指针、单调栈）；每个值必须对应一个真实存在的 `algorithms/<套路名>.md`，否则校验 F 拦截。
- **手写的只有套路本身的讲解**：一句话定义 + **"为什么这些题是同一个套路"**（唯一有智力增量、脚本写不出来的部分），其余按套路性质取舍复杂度/易错点/对比同类几节。
- **两级归属**：`topics:` 是粗套路（13 个，决定题目出现在哪个套路页）；`techniques:` 是细技术词（44 个，决定题目在套路页里落到哪个分组）。两者都必填，校验 F / S 分别拦截。
  - 词表的权威源是**套路页自己的 frontmatter `techniques:`**，13 页的并集即全局词表。新增技术词 = 在对应套路页声明。
  - **同一个词可以被多个套路页声明**。白名单的语义是"这一页展示哪些分组"，不是"这个词归谁所有"。`哈希查表` 同时被哈希表、数组与字符串、字典树三页声明——这正是"`01-两数之和` 挂在数组页下，但它的技术纯粹是哈希"这类**挂靠型 topic** 的解法：数组只是它作用的数据类型，不是技术来源。
  - **`techniques` 以「代码实际实现的解法」为准，不是「思路里提到的解法」**。检索标签的契约是：按标签找过来，能看到对应的代码。`720-词典中最长的单词` 思路首列 Trie 但代码是排序+哈希集，故标 `哈希查表` 而非 `Trie前缀树`。
  - `模拟构造` 是兜底词（按题意直接模拟、没有可归纳的技术）。**覆盖数超过 8 题就要复查**，说明有题被偷懒归类了。
  - 词表成色由 `gen_topics.py` 打印审计（仅覆盖 1 题的词会高亮），**不做硬校验**。曾试过「每词 ≥N 题」的阈值：N=3 逼出 `快速选择`→`堆TopK`、`BST中序`→`递归返回值设计` 两个语义上就错的合并；降到 N=2 后 `快速选择` 自己又只剩 1 题。词表条数不是成本（回填才是），拿条数下限去卡语义精确是拿错了东西当约束。
- **成员列表绝不手写**：`## 已解题目` 由 `python3 scripts/gen_topics.py` 从全库题解的 `topics:` / `techniques:` frontmatter 生成，按技术词分 H3 组，校验 R 拦过期——这是套路节点版本的"视图永远生成，绝不手写"（同 `概念/` 的 Q 项）。一个套路页只有在它的题**全部标注完**之后才切换成分组显示，否则整页保持平铺（避免"几个小组 + 一大坨未标注"）。
- **套路节点不是跨域概念**：套路节点只服务算法侧（不要求跨 `interview/`），不受校验 P 约束；如果某个套路的核心机制在 `interview/` 侧也有独立应用（如"摊还""局部性"），应该在套路页里链接对应的 `概念/` 节点，而不是把套路页本身也塞进 `概念/`。

## 跨分类概念规范（概念/，校验 P/Q 拦截）

`概念/` 放**跨 interview 分类的知识节点**——同一个机制在多个八股分类里反复出现，需要一个地方把它们接起来（如 `幂等` 串起消息队列、分布式事务、接口治理；`间接层` 串起代理、虚拟内存、路由）。

- **入场券 = 跨 interview 分类，不是"看起来重要"**：入链必须覆盖 ≥2 个 interview 分类，否则校验 P 判「未成体系」，应回收进正文。
  - 索引页/首页的**导航链接不计入**——否则自己链一下就凑够两个域，判据失效。
  - **`algorithms/` 也不计入**。算法侧有自己的概念层（13 个套路页，与 `概念/` 同构），让 `概念/` 去覆盖算法等于在已有概念层上再盖一层，见 CLAUDE.md 的「分域原则」。
  - 算法 → 概念 的**单向链接仍然允许**，也照常展示在 `## 出现在哪里` 里，只是不算入场券——**「可以链」和「必须链」是两回事**。不要为了让某个概念达标而去算法侧硬凑入链。
- **手写的只有两样**：一句话定义，以及**「为什么这些看起来无关的东西是同一个模式」**。后者是概念页唯一有智力增量、脚本写不出来的部分。
- **成员列表绝不手写**：`## 出现在哪里` 由 `python3 scripts/gen_concepts.py` 从全库真实入链生成，校验 Q 拦过期。手写的成员列表最终都会变成漂移的 `高频题目索引.md` A 表。
- **按语义聚合，不按字面**：`一致性哈希` 的"一致性"与 CAP 的 consistency 是两回事；`批处理摊薄固定开销`（分摊固定成本）不是 `摊还`（偶发昂贵操作被廉价操作预付）。同名不同义的必须分开，并在页里立 `## 易混:` 一节点破。
- **聚合页要带锚点**：概念页若下辖多个子概念（`树` 含 BST/平衡树/B+树），引用方应链到具体锚点 `[树](树.md#b-树)`。Quartz 的 backlinks 是**文件级**的（`crawl-links` 存边时丢弃锚点），锚点级溯源只能靠 `gen_concepts.py` 自己解析 markdown 得到。
- **内容单一归属，引用可多处**：机制讲一处（`树.md#b-树` 讲 B+ 树为什么矮、叶子为什么串链表），选型讲另一处（`MySQL.md` 讲 MySQL 为什么选它）——这是两个知识点，不是重复。

## 修改与整合规范

- **重复考点**：一处深讲，他处一句话 + 链接。选"更系统/更深"的一方做权威（如会话安全归安全认证、CAP 归分布式系统）
- **跨域视角不同的重复**（死锁之于 Java/OS、零拷贝之于 OS/Kafka）：**保留两份，只加互链**，不删内容

## 存量内容整改规范（docsify 遗留内容升级，2026-07 启动）

迁移自 docsify 时期的旧内容与新体系并存，整改按以下规则执行（每篇一次整改批次，登记在 TODO.md）：

1. **章 = 主题，不是合并批次**：消灭"`## 一、XXX` + `## 二、XXX 补充`"式拼接——"补充"章的小节按主题归入正式章节，章号只表达阅读顺序。同一考点在一篇内只出现一次（跨篇重复走「修改与整合规范」）。
2. **拆掉 docsify 三件套外壳**：`核心概念速查` / `高频面试题` / `最佳实践` 这层壳不保留——速查表无增量信息则删，有信息则并入对应小节；壳下的问题小节提级为正常 H3。
3. **浅小节升级**：只有"快答"两三行的小节，升级为知识点模板结构（至少补"为什么"和"常见追问"各一条）；暂时无力深写的保持简短，但登记 TODO，不留"深答省略/详见本文档"式的空头引用。
4. **事实基线（版本口径）**：
   - 全站以 **JDK 8 为对照基线、17/21/25 为现代口径**，提及版本敏感的实现细节必须标注版本（如 String 的 char[]→byte[] 是 JDK 9 分界）
   - 预览/孵化特性必须写明 JEP 状态与落地版本，**禁止把未落地提案写成已发布**
   - 工具/库推荐避开有安全争议或已停维护的选择（如 fastjson 1.x → Jackson/fastjson2）
5. **追问地图 = 正文目录**：地图每行的"必讲关键点/下一层追问"必须能在正文找到承接小节；正文没有的，要么先补正文，要么该行暂缓上地图（缺口登记 TODO）。
6. **元数据行补频次**：旧式只有 `难度 🟡` 的行合法但信息量不足，整改时对 ★★★★ 以上的高频小节补齐 `频次` 段；把握不准的保持 `难度` 单段，不硬编。
